## 一、评审议程

### 评审目标

确认以下事项，使PRD和原型达到可进入开发的状态：

| # | 评审项 | 负责人 | 产出 |
|---|--------|--------|------|
| 1 | 设计方案是否完整覆盖所有页面和状态 | 设计师 | 页面清单确认、交互边界确认 |
| 2 | 技术方案是否可行、有无阻断风险 | 技术负责人 | 技术选型确认、架构方案确认 |
| 3 | 接口设计是否清晰、数据结构是否完整 | 后端 | API文档草案、数据库表结构 |
| 4 | 验收标准是否可测试 | 测试 | 测试用例框架 |
| 5 | 待确认问题的决策 | 全体 | 决策记录 |

### 评审流程建议

```
第一部分：产品宣讲（15分钟）
  ├── 产品背景与目标
  ├── 核心用户流程演示（原型走查）
  └── MVP功能范围确认

第二部分：设计评审（20分钟）
  ├── 9个页面逐一走查
  ├── 状态清单确认
  ├── 交互边界讨论
  └── 设计验收点确认

第三部分：研发评审（25分钟）
  ├── 技术架构方案
  ├── 接口设计讨论
  ├── 数据结构确认
  ├── 实现风险识别
  └── 待确认技术问题决策

第四部分：测试评审（10分钟）
  ├── 验收标准确认
  ├── 边界场景补充
  └── 测试策略对齐

第五部分：总结与待办（10分钟）
  ├── 记录所有待确认问题及负责人
  ├── 确定下一步排期
  └── 分配待办事项
```

## 二、设计评审：页面清单与交互边界

### 2.1 完整页面清单（9个页面）

| # | 页面 | 类型 | 核心元素 | 交互要点 | PRD引用 |
|---|------|------|----------|----------|---------|

| P2 | 场景选择面板 | 浮层Sheet | 2个场景卡片（腹泻急/带娃急） | 点击→对应模式推荐 | F-01 |

### 2.2 交互边界讨论项

| # | 交互场景 | 当前方案 | 需讨论 |
|---|----------|----------|--------|
| 1 | 首页"急"按钮的长按触发 | 长按≥500ms展开场景面板 | 移动端长按是否会被系统手势拦截？是否需要替代方案（如双击）？ |
| 2 | 推荐结果页上滑展开备选 | 上滑卡片展开备选列表 | 上滑手势与地图缩放是否冲突？需要明确手势区域划分 |
| 3 | 反馈弹窗触发时机 | 用户离开厕所范围(≥100m)或返回小程序时弹出 | 小程序在后台时无法检测位置变化，如何判断"离开范围"？建议改为：用户从导航返回小程序时弹出 |
| 4 | 室内厕所导航终点 | 导航到建筑入口+文字指引 | wx.openLocation只能传一个坐标，文字指引如何展示？建议：导航前先展示覆盖层，用户点击"开始导航"后再跳转 |
| 5 | 地图标记点击 | 点击厕所标记→详情页 | 首页地图标记点击后是直接进详情页还是先展示简要信息？建议：先展示底部简卡片，再点击进详情 |
| 6 | 场景选择面板关闭 | 点击关闭按钮或面板外区域 | 是否需要遮罩层？建议：有半透明遮罩，点击遮罩关闭 |
| 7 | 多次跳过反馈 | 不强制，降低弹窗频率 | 具体策略：每3次到达弹1次？还是连续跳过2次后不再弹？需产品确认 |
| 8 | 分享卡片展示 | 卡片标题含厕所名+状态+步行时间 | 步行时间对接收者无意义（位置不同），建议改为"距你X米"或去掉步行时间 |

## 三、设计评审：状态清单

### 3.1 厕所状态（三色灯）

| 状态 | 触发条件 | 视觉表现 | 可信度 |
|------|----------|----------|--------|
| 🟢 可用（高可信） | 最近5分钟内有"可用"反馈 | 绿色圆形● + "可用" | 95% |
| 🟢 可用（中可信） | 最近1小时内有"可用"反馈，超5分钟 | 绿色圆形● + "可用" | 60-90% |
| 🟡 存疑 | 最近反馈超1小时但≤3小时；或超3小时无"不可用"报告 | 黄色三角形▲ + "待确认" | 40-60% |
| 🔴 不可用 | 1小时内≥3人报告"不可用"；或最近反馈为"不可用"可信度≥85% | 红色方形■ + "不可用" | ≥85% |
| ⚪ 数据过期 | 超过6个月无反馈 | 灰色圆形 + "数据可能过期" | — |

> **⚠️ 设计确认点**：5种状态需要5种视觉表现，设计师确认是否需要"数据过期"单独一种视觉？还是复用🟡存疑+文字标注？

### 3.2 页面状态清单

| 页面 | 状态 | 触发条件 | 表现 |
|------|------|----------|------|
| 首页 | 正常 | 位置已授权，附近有厕所 | 地图+标记+"急"按钮可点击 |
| 首页 | 未授权定位 | 用户未授权位置 | 地图空白+定位引导按钮+"急"按钮置灰 |
| 首页 | 定位失败 | GPS获取失败 | "定位失败，请检查GPS设置"+重试按钮 |
| 首页 | 无数据区域 | 1000米内无厕所 | "当前区域暂无数据，欢迎贡献"+贡献入口 |
| 推荐结果 | 正常 | 有推荐结果 | 展示最佳推荐卡片 |
| 推荐结果 | 扩大搜索 | 500米无结果，扩大至1000米 | "附近厕所较少，已扩大搜索范围"提示 |
| 推荐结果 | 全部不可用 | 所有厕所状态为🔴 | 推荐最近的🟡，标注"附近厕所可能不可用" |
| 推荐结果 | 商场闭店 | 当前时间在营业时间外 | 厕所标记降为🟡，标注"可能已闭店" |
| 导航 | 正常 | 导航启动成功 | 展示路径指引 |
| 导航 | 导航失败 | wx.openLocation调用失败 | "导航启动失败"+重试按钮 |
| 反馈 | 正常 | 用户到达后 | 弹出4道选择题 |
| 反馈 | 位置不匹配 | 用户距厕所≥500m反馈 | 仍然允许反馈，但系统降低反馈权重 |
| 反馈 | 数据不足 | 质量画像少于3条反馈 | "数据较少，欢迎贡献" |
| 贡献 | 新用户 | 前5条提交 | 提交后提示"审核中"（强制审核） |
| 贡献 | 坐标异常 | 坐标落在道路中间 | "坐标似乎不在建筑内，请确认" |

## 四、设计评审：设计验收点

### 4.1 视觉规范验收

| # | 验收项 | 规范 | 验收方法 |
|---|--------|------|----------|
| 1 | 主色调 | 橙红#FF6B35（按钮）、深蓝#1A73E8（地图/链接） | 取色器验证 |

| 6 | 动效 | 仅状态转换（加载→结果），无装饰动画 | 原型走查 |

### 4.2 HIG原则验收

| # | HIG原则 | 验收项 | 验收方法 |
|---|---------|--------|----------|
| 1 | Clarity | 首页只有1个核心操作（"急"按钮） | 首屏截图检查 |
| 2 | Deference | 地图为底，操作浮层在上 | 首屏视觉层级检查 |
| 3 | Depth | 首页→推荐→导航，层级清晰 | 走查跳转路径 |
| 4 | Progressive Disclosure | 场景选择通过长按展开，不在首页铺开 | 首屏无场景选择入口 |
| 5 | Minimal Input | 默认智能推荐，无需选择；反馈用选择题 | 用户流程走查 |
| 6 | Feedback | 点击"急"立即显示加载状态；反馈提交即时确认 | 交互测试 |

### 5.2 技术选型确认

| 层面 | 方案 | 状态 | 备注 |
|------|------|------|------|
| 前端 | 微信小程序原生 | ✅ 已确认 | 不用框架，原生开发 |
| 地图 | 微信小程序map组件 + 腾讯位置服务SDK | ✅ 已确认 | 步行路线用SDK的direction(walking) |
| 导航 | wx.openLocation拉起外部地图App | ✅ 已确认 | 微信无内置导航 |
| 后端 | Node.js云函数 | ✅ 已确认 | 微信云开发或独立服务器 |
| 数据库 | PostgreSQL + PostGIS | 🔧 待确认 | vs 微信云开发数据库（见待确认问题1） |
| 缓存 | Redis | 🔧 待确认 | 用于热门厕所状态缓存 |
| AI推荐 | 规则引擎（权重矩阵） | ✅ 已确认 | V1不用ML |
| 审核后台 | Web端独立页面 | 🔧 待确认 | 技术栈待定（见待确认问题5） |
| 部署 | 微信云开发 or 腾讯云 | 🔧 待确认 | 取决于数据库选型 |

### 5.3 外部依赖

| 依赖 | 用途 | 风险 | 备选方案 |
|------|------|------|----------|
| 微信小程序地图组件 | 地图展示、标记点 | 无（原生能力） | — |
| 腾讯位置服务SDK | 步行路线规划、逆地址解析 | 个人开发者1万次/日免费配额 | 升级企业配额 |
| wx.openLocation | 拉起外部地图导航 | 无（原生API） | — |
| wx.getLocation | 获取用户位置 | 需用户授权 | 用户拒绝授权时引导手动选位置 |
| 微信分享能力 | 厕所链接分享 | 无 | — |
| 政府/OSM公厕数据 | 冷启动位置数据 | 数据格式不统一 | 人工录入替代 |

### 5.4 第三方API调用链

```
用户打开小程序
  │
  ├── wx.getLocation() → 获取用户经纬度
  │
  ├── 腾讯位置服务 SDK
  │   ├── qqmapsdk.reverseGeocoder() → 逆地址解析（展示位置名称）
  │   └── qqmapsdk.direction({mode:'walking'}) → 步行路线规划
  │
  ├── 后端API（云函数）
  │   ├── /api/toilet/nearby → 搜索附近厕所
```

### 7.2 地理位置查询方案

| 方案 | 说明 | 优劣 |
|------|------|------|
| **PostGIS ST_DWithin** | `SELECT * FROM toilets WHERE ST_DWithin(geom, ST_Point(lng,lat)::geography, 500)` | 功能强大，支持精确地理查询。需自建PostgreSQL服务器 |
| **微信云开发 geoNear** | `db.collection('toilets').where({location: db.command.geoNear({geometry, maxDistance: 500})})` | 简单易用，无需服务器。但功能有限制 |
| **手动Haversine公式** | SQL中计算距离 | 通用但性能差，不适合大量数据 |

> **🔧 待确认**：方案1 vs 方案2的选择，直接影响整体部署架构。

### 7.3 状态计算逻辑（伪代码）

```javascript
function calculateToiletStatus(toilet, feedbacks) {
  const now = Date.now();
  const recentFeedbacks = feedbacks
    .filter(f => f.toilet_id === toilet.id)
    .sort((a, b) => b.created_at - a.created_at);

  // 规则1：1小时内≥3人报告不可用 → 🔴
  const recentUnavailable = recentFeedbacks.filter(
    f => f.status === 'unavailable' && (now - f.created_at) < 3600000
  );
  if (recentUnavailable.length >= 3) {
    return { color: 'red', confidence: 85 };
  }

  // 规则2：最近5分钟内有"可用"反馈 → 🟢 95%
  const latestAvailable = recentFeedbacks.find(f => f.status === 'available');
  if (latestAvailable && (now - latestAvailable.created_at) < 300000) {
    return { color: 'green', confidence: 95 };
  }

  // 规则3：最近1小时内有"可用"反馈 → 🟢 60-90%
  if (latestAvailable && (now - latestAvailable.created_at) < 3600000) {
    const decay = 90 - (now - latestAvailable.created_at) / 300000 * 30;
    return { color: 'green', confidence: Math.max(60, decay) };
  }

  // 规则4：超过3小时或无反馈 → 🟡
  if (!latestAvailable || (now - latestAvailable.created_at) > 10800000) {
    return { color: 'yellow', confidence: 40 };
  }

  // 规则5：1-3小时 → 🟡
  return { color: 'yellow', confidence: 50 };

  // 特殊规则：商场闭店、地铁末班车等需额外检查operating_hours
}
```
