## 二、不一致项详细清单

### 2.1 参数不一致

#### ⚠️ [1] GET /toilets/nearby — 前端未传递完整筛选参数

**后端支持参数**：`lat`, `lng`, `radius`, `type`, `status`, `has_baby`, `has_paper`, `page`, `page_size`

**前端调用**（`pages/index/index.js:56-59`）：
```js
api.getNearbyToilets({ lat: loc.lat, lng: loc.lng })
```
仅传了 `lat` 和 `lng`，未使用 `radius`、`type`、`status`、`has_baby`、`has_paper`、`page`、`page_size` 等参数。

**影响**：首页加载时无法按条件筛选，默认行为可行但功能不完整。

**修复建议**：
- 首页添加筛选栏控件，支持类型、状态、半径选择
- 列表页支持分页加载（目前Mock一次性返回全部）

---

#### ⚠️ [2] POST /recommend — 前端缺少 `radius` 和 `top_k` 参数

**后端请求体**：
```json
{ "lat": 30.658, "lng": 104.082, "scene": "smart", "radius": 2000, "top_k": 5 }
```

**前端调用**（`pages/index/index.js:134-138`）：
```js
api.recommendToilet({ scene, lat: loc.lat, lng: loc.lng })
```
缺少 `radius` 和 `top_k` 参数。

**影响**：后端使用默认值（radius=2000, top_k=5），目前不影响功能。但如果后续需要自定义搜索半径或返回数量，需要补充。

**修复建议**：
- 在 `recommendToilet()` 调用中补传 `radius: 2000, top_k: 5`，至少显式传默认值
- 如果未来支持半径自定义，需要添加UI控件

## 四、整体结论

### 对齐度评估：**约 20%**

15个后端API中，前端仅对接了 5 个（其中仅 1 个完全对齐，4 个存在参数或响应结构问题）。

### 主要问题

1. **前端Mock数据与后端响应结构差异大**：Mock采用扁平+冗余设计，后端采用嵌套结构，接入真实API后多处取值会返回 `undefined`。

2. **核心流程缺失**：登录认证、用户信息获取这两个基础接口前端均未实现调用。

3. **前端自创了3个后端不存在的接口**：导航步骤、贡献厕所、快速确认状态，需协调后端补充。

4. **参数传递不完整**：部分接口前端只传了必填参数，未利用后端的筛选/分页能力。

### 建议行动

| 阶段 | 内容 | 预估工作量 |
|------|------|-----------|
| 第一阶段（阻塞） | 补齐 `api.js` 中缺失的10个函数 + 修复登录流程 | 1-2天 |
| 第二阶段（阻塞） | 对齐Mock数据结构与后端响应格式 | 0.5天 |
| 第三阶段 | 适配前端页面中的字段读取路径 | 0.5天 |
| 第四阶段 | 后端补充3个缺失接口 / 前端调整调用方案 | 协商 |
| 第五阶段（优化） | 引入数据适配层，增加筛选/分页UI | 1-2天 |

### 风险提示

- **高风险**：当前Mock数据同时保留了扁平和嵌套字段，前端代码混合使用两种路径取值。接入真实后端后，如果只保留嵌套结构，**至少有3个文件（detail.js, share.js, index.js）** 需要修改字段访问路径。
- **中风险**：`getNearbyToilets()` 返回了整个Mock对象（包含详情专属字段），如果首页逻辑依赖了详情字段，切换真实API后会出错。
- **低风险**：推荐接口返回的 `context` 对象（`is_night`, `is_meal_time`, `in_safe_zone`）前端未使用，暂不影响功能。
