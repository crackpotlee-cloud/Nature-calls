# 急厕 · 产品逻辑设计

> 不讲功能规格。先讲清楚产品为什么成立、与竞品的根本差异是什么、每一层逻辑怎么推出来的。

---

## 一、产品哲学：三个不该由用户回答的问题

用户打开任何地图App找厕所时，其实在脑子里问了三个问题：

1. **厕所在哪？** → 地图能回答，但只回答了一半（回答了建筑位置，没回答入口位置）
2. **能用吗？** → 地图回答不了
3. **哪个最好？** → 地图完全没有这个维度

而真正的需求是：**我现在很急，告诉我应该去哪个**。

这不是"搜索"，是"推荐"。

这不是"找到POI"，是"找到资源"。

这不是"静态数据"，是"实时信号"。

**根本差异**：

| | 地图App | 急厕 |
|---|---|---|
| 解决什么问题 | 厕所在哪 | 我应该去哪个 |
| 数据模型 | POI（静态位置） | Resource（动态资源） |
| 排序逻辑 | 直线距离 | 到达时间 × 可用概率 × 质量评分 |
| 用户行为 | 搜索→选一个→导航 | 一键→系统推荐→导航 |
| 数据来源 | 商业采集+政府数据 | 用户众包+AI校验 |
| 护城河 | POI数量 | 实时状态数据 + 空间关系知识图谱 |

**这不是"地图+厕所标签"，这是一个厕所的实时资源调度引擎。**

## 二、六层产品逻辑

### 第一层：空间位置关系 — 不是"在哪里"，是"怎么找到门"

#### 为什么地图的路子走不通

地图POI只记录一个坐标点。对大多数POI类型，一个坐标足够——餐厅、加油站、便利店，导航到点就行。

厕所是少数**"坐标点≠目的地"**的POI类型。

- 室外厕所：坐标点可能是建筑中心，但入口在建筑背面。你导到正面就懵逼了。
- 室内厕所：坐标点可能是商场中心，但厕所在3楼电梯口后面。你到了商场还是找不到。

#### 产品逻辑：建立"空间参照物关系"，不是"坐标点"

每一个厕所的数据结构不是 `(lat, lng)`，而是 `(reference_point, spatial_relation)`。

**室外厕所的结构：**

```
XX公园公厕
├── 入口位置：公园南门进去右手边50米
├── 男厕位置：入口左侧
├── 女厕位置：入口右侧
├── 无障碍位：入口正前方
└── 导航终点设置为"入口"，不是"建筑中心坐标"
```

**室内厕所的结构（这才是差异化的关键）：**

```
XX购物中心
├── 3F 海底捞旁边 → 从3号电梯上3楼，出来右转，海底捞门口旁通道进入
├── 5F 电影院入口右侧 → 乘直梯至5F，电影院售票厅右侧走廊
├── B1 超市出口旁 → B1扶梯下楼后，超市收银台出口对面
└── 每层都有一个，但用户应该被引导到"离入口最近的"那个
```

**为什么要用"海底捞旁"而不是"3F走廊"？**

因为大型商家的指引系统更完善。你在商场里找"3F走廊"很难，但找"海底捞"很容易——商场里的导视牌、楼层索引都会标注海底捞在哪。用海底捞、电影院、星巴克、优衣库这种**导航基础设施最完善的品牌**作为"锚点"，间接帮用户找到厕所。

这是借用了"找路心理学"：人不是靠坐标找路的，是靠地标找到地标再找到目的地。

#### 导航终点重定义

最关键的改动：**导航终点不是建筑中心，是厕所入口**。

这需要数据层面的重构：
- 将建筑的导航坐标标记为"最近的可达入口"（不是商场大门，是离厕所最近的那个门）
- 如果厕所不在建筑外围，终点设在"建筑的厕所所在楼层通道入口"
- 如果需要过安检（地铁站），终点设在"安检口"，并标注"过了安检左转50米"

### 第二层：可用性状态 — Waze的"实时路况"逻辑

#### 为什么"存在"和"可用"是两个完全不同的概念

地图POI告诉你"这里有一个厕所"。但这就像告诉你在路上"这里有一条路"——有用，但不够。

你需要知道的是"这条路现在通不通"。

#### 产品逻辑：三色信号灯 + 时效衰减 + 人类可读的证据链

**不是简单的"🟢 能用 / 🔴 不能用"**，而是带时间戳和可信度的信号：

```
🟢 王先生 2分钟前确认可用 · 可信度 95%
   今天已有 8 人确认正常
   
🟡 最近一次确认是 3小时前 · 可信度 60%
   上次3人确认可用，之后无人反馈
   可能正常，也可能刚关了

🔴 李先生 15分钟前报告维修中 · 可信度 85%
   过去1小时内 3 人报告不可用
   建议选择替代厕所
```

**数据模型的核心设计：**

1. **每次反馈是一票，不是覆盖**。10个人说"能用"+ 1个人说"关了" ≠ "关了"。是"10票可用，1票关停，可能刚刚关，建议谨慎"。

2. **时效衰减不是线性的**。不同场景不同衰减速度：
   - 商场厕所：营业时间内的反馈有效，闭店后所有反馈自动降为🟡
   - 地铁厕所：末班车后反馈无效（因为关闭状态不是"不可用"，是"正常运营结束"）
   - 公共厕所：24小时开放型，反馈24小时后开始衰减

3. **"没有反馈"本身也是信号**。如果一个热门商圈的厕所超过6小时无人反馈，比冷门地段48小时无反馈更可疑——因为热门区域理论上应该有更多用户经过。

#### 信任机制：不是匿名，是隐名

用户反馈时不需要注册、不需要填信息。但每条反馈背后都有一个系统自动建立的信任评分：
- 这个人的历史反馈准确率（他之前说"能用"的厕所，后来别人也说能用 → 他的可信度↑）
- 反馈时的位置与厕所位置是否匹配（在500米外反馈"这个厕所能用" → 权重↓）
- 反馈时间是否在厕所的合理使用时间段内

### 第三层：厕所质量 — 为什么"能用"不够

#### 用户心理的真相

用户嘴上说"我只要一个厕所就行"，但实际上选择逻辑是：

> 如果步行3分钟有个脏的，步行6分钟有个干净的 → 大多数人会选哪个？

答案是**分场景**：
- 拉肚子 → 脏的也行，越快越好
- 普通小便 → 可以忍一忍找干净的
- 带小孩 → 必须干净的 + 有母婴台的
- 女生 → 干净优先于距离

#### 产品逻辑：四个维度的质量画像

不是"评分"，是"画像"。评分会掩盖信息（一个4.2分的厕所什么都说明不了），画像让用户自己判断。

| 维度 | 衡量方式 | 数据来源 |
|------|----------|----------|
| 🧹 清洁度 | 最近N人评价"干净/一般/脏"的分布 | 用户选择题（不用打分，避免打分偏差） |
| 👥 排队情况 | 最近N小时反馈"排队/正常/空" | 用户反馈 + 时段关联（午饭后排队概率高） |
| 🧻 纸巾供应 | 最近反馈"有纸/无纸"的最新票 + 历史规律 | 用户反馈 |
| 🔧 设施完整度 | 洗手液/烘手机/母婴台/无障碍位/空调 | 结构化录入为主，用户可补充 |

**为什么不用5星评分？**

因为5星评分有文化偏差（有人觉得干净就给5星，有人觉得"正常水平"就给3星）、有动机偏差（恶意差评）、有时间偏差（去年干净不代表今年干净）。

用**选择题**代替评分：
- "干净程度：我今天愿意在这里多待一会儿 / 能用 / 不太行"
- "排队：我前面没人 / 等了1-2人 / 排长队"
- "有纸吗：有 / 只剩几张了 / 完全没有"

选择题的答案分布比评分更能反映真实情况。"最近10个人里有7个选了'愿意多待一会儿'"比"评分4.2"更有信息量。

### 第四层：AI推荐 — 从"搜索"到"决策"

#### 核心洞察

现有产品是"我给你一个列表，你自己选"。
急厕应该是"我告诉你应该去哪个，然后告诉你怎么去"。

这不是界面优化，是产品定位的根本转变。

#### AI在做的是四维决策融合

当用户点"急"按钮时，系统不是在搜索——是在做决策。综合考虑：

```
推荐分数 = 
    (1 / 到达时间权重) × 
    可用概率 × 
    质量适配度 × 
    场景匹配度
```

**到达时间**：不只是距离。包括：
- 步行距离
- 是否需要过马路（过马路+30秒）
- 室内厕所额外的时间（商场：入口到厕所平均2分钟）
- 地铁安检时间（通常+2分钟）

**可用概率**：基于反馈时效衰减计算，不是二值。

**质量适配度**：不同场景质量权重不同。拉肚子场景清洁度权重低，带娃场景母婴台权重极高。

**场景匹配度**（见第五层）。

#### 推荐结果的呈现

不是"3个选项给用户选"。而是：

1. 首屏显示**1个最佳推荐**，大字："步行3分钟 · 最近5分钟有人确认可用 · 免费进入"
2. 如果有疑问，滑一下看备选
3. 底部一个大按钮"导航"

**用户不需要思考，只需要行动。**

### 第五层：应急模式 — 不同急法，不同解法

#### 为什么"急"要分场景

"尿急"和"拉肚子"和"带小孩上厕所" 是三个完全不同的问题：

| 场景 | 最在意的 | 可以容忍的 | 绝对不能接受的 |
|------|----------|-----------|---------------|
| 💧 小解急 | 速度 | 环境一般 | 排长队 |
| 🏃 腹泻急 | 速度 + 有纸 + 隐私 | 一切 | 没纸 |
| 👶 带小孩 | 安全、母婴台 | 距离远一点 | 脏、没有母婴台 |
| 🤰 孕妇 | 安全、干净、不排队 | 距离远一点 | 楼梯、蹲坑 |
| 👴 老人 | 有无障碍、坐便 | 距离远一点 | 蹲坑、楼梯 |
| 👩 女性用户 | 干净、安全、不排队 | 距离远一点 | 脏、排队 |

#### 产品逻辑：同一套数据，不同推荐逻辑

不同场景，AI推荐算法中的权重矩阵完全不同：

**小解急模式**：到达时间权重 = 90%，质量权重 = 10%
→ "别管干不干净，最快能上的"

**腹泻模式**：到达时间权重 = 50%，纸巾保证权重 = 40%，隐私性权重 = 10%
→ "最快的 + 确保有纸的"，且推荐隔间多/人流少的

**带娃模式**：母婴台 → 硬筛选条件（没有直接排除），清洁度权重 = 60%
→ "有母婴台的、最干净的"

**女性模式**：安全区权重 ↑，排除偏僻/停车场/夜间人少的位置

#### 进入方式

不是让用户在菜单里选（急了谁有耐心选）。而是：
- **默认快捷键**：首页"急"按钮 = 小解模式（最高频）
- **语音输入**："我拉肚子"、"带小孩上厕所"（最直觉）
- **微信聊天框**：收到"厕所"消息时，长按 → 发送给急厕小程序

### 第六层：把"非公厕"纳入体系 — 打破"公厕"的思维边界

#### 一个反常识的事实

中国大部分城市，**真正好用的厕所不在"公厕"里，在商业空间里**。

- 麦当劳/肯德基的厕所：24小时（部分）、干净、有纸、有洗手液、免费
- 星巴克的厕所：不需要消费（大多数门店）、环境好
- 商场厕所：免费、多层分布、设施齐全
- 酒店大堂厕所：高端酒店随便进、干净、几乎不排队
- 加油站的厕所：快速可达、基本都有

但传统地图把这些和"公共厕所"分成了不同的POI类别。

#### 产品逻辑：用户不在乎厕所归谁，只在乎能进吗

厕所的"归属"字段对用户没意义。有意义的是"准入成本"：

| 准入类型 | 说明 | 用户感受 |
|----------|------|---------|
| 🆓 自由进入 | 推门就进，无需任何条件 | 最好 |
| 🛂 需过安检 | 地铁站/高铁站（买票或安检才能进） | 可接受但麻烦 |
| ☕ 建议消费 | 麦当劳/星巴克（理论上要消费，实际一般不管） | 稍微有点心理负担 |
| 🛍 必须消费 | 餐厅内部厕所（不消费不让进） | 有成本 |
| 🔑 需门禁 | 写字楼/办公楼（刷卡才能进） | 基本进不去 |
| 🔢 需密码 | 部分商家厕所密码锁 | 需要问店员 |
| 📋 需登记 | 部分场所需要登记 | 很麻烦 |

#### "建议消费"的厕所在产品里的特殊处理

这是最微妙的一类。麦当劳厕所理论上是顾客的，但实际上几乎没人管。

产品策略：
- 标注为"建议消费（多数人直接进无问题）"
- 用户反馈会更新这个判断：如果陆续有人反馈"被拦了"，准入类型自动降级为"需消费"
- 如果长期无人反馈被拦，维持"建议消费"状态

这本身就是一个持续更新的知识。

## 三、把六层串起来：一个完整的用户旅程

### 场景：下午2点，你在春熙路逛街，突然尿急

**第一秒**：打开微信 → 下拉 → 急厕小程序（0.5秒）

**第二秒**：看到首页大按钮"急！找厕所"（不需要思考点哪里）

**第三秒**：不用点搜索，不用打字。AI已经在后台完成了：
- 获取当前位置：春熙路IFS附近
- 扫描周边500米内的厕所（不仅是公厕，还包括IFS、太古里、麦当劳、星巴克、屈臣氏……）
- 每个厕所检查：最新可用状态、准入条件、质量评分
- 根据你的使用历史：你是男的 → 不推荐女厕排队少的选项；你之前选过"干净优先"→ 质量权重↑
- 融合计算 → 推荐最优

**第四秒**：屏幕显示：
```
🥇 IFS 3F 海底捞旁
步行3分钟 · 5分钟前可用 · 免费进入
干净 👍 排队少 👍 有纸 👍
[ 开始导航 ]
```

**第五秒**：点导航。导航终点不是IFS大门口，是"3号直梯口"。
到电梯口了 → 提示"电梯上3楼，出来右转，海底捞门口旁通道"
到了 → 提示"就是这里，厕所在你左边"

**上完厕所后**：App弹一个轻量反馈
```
这个厕所：
[ ✅ 能用 ] [ ❌ 不行 ]
你愿意贡献3秒帮下一个急的人吗？
```

用户顺手点一下✅，走了。

这就是完整的闭环：**5秒决策 + 精准导航 + 3秒反馈 = 下一个人的数据更新。** 没人需要做多余动作。

## 四、Waze模式的数据飞轮

### 飞轮运转逻辑

```
用户急 → 打开急厕 → 一键推荐 → 到达厕所
                                      ↓
                              顺手反馈（3秒选择题）
                                      ↓
                              厕所状态数据更新
                                      ↓
                         下一个用户搜到这个厕所时更准
                                      ↓
                         更多人来用 → 更多反馈
                                      ↓
                              数据越来越准
                                      ↓
                         比任何竞品都更"懂"这个厕所
                                      ↓
                              护城河加深
```

### 冷启动时飞轮怎么转起来

**鸡生蛋问题**：没有用户就没有数据 → 没有数据用户就不来。

破解方案（四管齐下）：

**1. 政府/OSM底座数据提供"位置"**（免费、静态）
→ 解决"厕所在哪"的初始数据，不需要用户贡献

**2. 种子用户人工录入"空间关系"**（核心差异化数据，需要人工）
→ 找10-20个人，每人录入自己最熟悉的20个厕所的"怎么找门"信息
→ 外卖骑手知道每个商场/餐厅厕所的精确位置，是天然的种子贡献者
→ 一个城市录完200-400个核心厕所，覆盖地铁站+商圈+公园足够了

**3. 用"帮我确认"机制冷启动反馈数据**
→ 第一批用户在厕所里，看到"这个厕所的状态还是3天前的，你愿意花3秒帮确认吗？"
→ 不需要"写评价"，就点三个✅/❌
→ 因为需求本身是刚需（在厕所门口打开App正是最相关的时候），转化率会远高于普通UGC产品

**4. 精确导航本身就是拉新手段**
→ 有人微信分享一个厕所的精确导航链接："IFS 3楼海底捞旁这个厕所有纸"
→ 接收者点开链接 → 直接导航到厕所 → 等于一次产品体验
→ 不需要下载App，不需要注册，小程序原生体验

### 为什么这能形成壁垒

地图公司有千万POI，但没有：
- 每个厕所的"空间参照物关系"（海底捞旁、电影院右侧）
- 每个厕所的"实时可用状态"（社区众包）
- 每个厕所的"质量画像"（清洁度、排队、设施）
- 每个商业厕所的"准入条件动态知识"（能不能直接进）
- 每个厕所的"导航终点"（入口不是建筑中心）

这些数据不是"一次性收集"可以解决的，是需要**持续社区维护**的——而这才是真正的护城河。地图公司不可能为厕所这个品类建立社区。

就像Waze的交通数据对Google Maps形成壁垒一样，急厕的厕所动态数据对高德百度形成壁垒。

## 五、MVP的产品边界：什么做，什么坚决不做

### 核心原则

**只做"替代不了的功能"**。如果地图App已有的功能（地图、导航、POI搜索），不用做——调用微信的就行。

急厕只做三件事：
1. **关系数据**：空间参照物 + 准入条件（地图没有，也不会做）
2. **状态数据**：实时可用性 + 质量画像（地图没有社区能力做）
3. **AI决策**：场景化推荐（地图的产品逻辑不支持）

### MVP功能范围

| 功能 | MVP | 理由 |
|------|-----|------|
| 一键AI推荐 | ✅ | 核心体验，产品灵魂 |
| 厕所空间参照物数据 | ✅ | 核心差异化，第一批人工录入 |
| 三色状态灯+时效衰减 | ✅ | Waze模式核心 |
| 轻量选择题反馈 | ✅ | 飞轮启动器 |
| 场景模式（小解/腹泻/带娃） | ✅ 先做3个 | 区分度最大的三个 |
| 质量画像（清洁/排队/设施） | ✅ | 厕所大众点评 |
| 非公厕准入标注 | ✅ | 这是数据的灵魂 |
| 导航终点=入口 | ✅ | 体验核心差异 |
| "帮我确认"分享 | ✅ | 冷启动+传播 |
| 用户积分/等级 | ❌ V1不做 | 先有数据再激励 |
| 历史记录/收藏 | ❌ V1不做 | 不是核心差异 |
| 照片/文字评价 | ❌ V1不做 | 增加数据治理成本 |
| 离线包 | ❌ V1不做 | 地铁无网场景后续考虑 |

## 六、关键待确认决策

这些需要在开发前定下来：

**决策1：产品名字**
- 急厕（直击痛点、有传播力、可能审核敏感）
- 厕达（强调差异化，更温和）
- 或者你有更好的想法？

**决策2：试点范围**
- 先做达州市区还是先从宣汉县开始？
- 宣汉县小，可控但数据量少；达州市区数据量多但录入工作量大

**决策3：应急模式的场景优先级**
- 首版做3个场景：小解急（默认）、腹泻急、带娃急
- 够不够？要不要加入"女性/夜间安全"场景？

**决策4：种子数据录入方案**
- 你自己一个人录，还是找朋友帮忙？
- 我可以设计一个录入模板，把过程标准化

**决策5：下一步**
- 先做交互原型（可点击的页面）验证体验？
- 还是直接进入系统架构设计+开发？
