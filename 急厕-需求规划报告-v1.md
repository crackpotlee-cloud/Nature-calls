## 一、竞品与行业分析

### 1.1 市场背景

中国公共厕所市场规模庞大。2024年中国公厕改造市场规模达 **587亿元**，预计2025年将攀升至 **678亿元**[1]。但现有市场投入集中在 **硬件改造**（物联网传感器、智能厕位、环境监测），**软件层面——帮用户找到并选择厕所——几乎是空白**。

智慧公厕方案主要面向B端（政府/商场管理者），提供厕位监测、环境监测、清洁调度等功能。普通用户端的小程序只是展示厕位空闲数，并不解决"应该去哪个"的决策问题。

### 1.2 现有竞品分析

| 产品 | 形态 | 核心功能 | 优势 | 劣势 |
|------|------|----------|------|------|
| **高德/百度地图** | App | POI搜索"厕所"，按距离排序，导航到坐标点 | 用户基数大、地图数据全、导航能力强 | 只有"位置"，没有"状态"；导航终点是建筑中心而非入口；不区分公厕/商业厕所 |
| **便了么** | App | 记录排便健康 + 找厕所 | 2018年上线，有先发优势 | 定位偏健康管理而非找厕所；需要付费会员；实时位置授权引发隐私担忧；找厕所功能弱 |
| **今厕一定得** | 微信小程序 | 查找洗手间、清洁度查看、AI语音、小游戏 | 2026年小程序创新大赛一等奖；高中生作品有传播话题性；功能较丰富 | 未正式上线；核心仍基于静态POI搜索；没有实时状态数据；推荐逻辑简单 |
| **AI厕急达 (ToiletGo)** | H5/WebView | 附近厕所搜索+智能排序+步行导航 | 开源项目；多维度评分排序；AI偏好解析；技术方案成熟可参考 | 纯技术Demo；无社区数据；无实时状态；依赖腾讯地图POI，数据质量受限于第三方 |
| **智慧公厕小程序** | 微信小程序 | 对接IoT设备，显示厕位空闲数 | 数据准确（硬件采集） | 仅覆盖安装了IoT设备的公厕（极少）；无质量评价；无推荐逻辑 |

### 1.3 竞品空白与急厕的差异化空间

```
现有产品覆盖的：  ✅ 厕所在哪（地图POI）
                 ✅ 厕位空闲（IoT公厕，极少覆盖）
                 ✅ 静态评分（今厕一定得的清洁度打分）

现有产品缺失的：  ❌ 厕所入口在哪（空间参照物关系）
                 ❌ 现在能用吗（实时社区反馈）
                 ❌ 哪个最好（场景化AI推荐）
                 ❌ 非公厕能不能进（准入条件知识库）
                 ❌ 带小孩/腹泻/夜间安全等细分场景
```

**急厕的定位不是"更好的厕所搜索"，而是"厕所实时资源调度引擎"**。这一差异化在现有竞品中完全没有被覆盖。

### 1.4 Waze类比验证

Waze的成功证明了三件事，同样适用于急厕：

1. **社区众包可以产生地图公司无法复制的实时数据**（交通路况 vs 厕所状态）
2. **轻量反馈机制可以维持数据飞轮**（被动上报路况 vs 3秒选择题反馈）
3. **专业化推荐比通用搜索更有价值**（Waze的路由 vs Google Maps的路由，前者的路况感知更强）

## 二、目标用户画像与场景矩阵

### 2.1 核心用户画像

| 画像 | 特征 | 核心痛点 | 使用频率 | 付费意愿 |
|------|------|----------|----------|----------|
| **都市通勤族** | 25-35岁，一二线城市，经常外出/逛街/出差 | 陌生区域找不到厕所，找到又怕脏/排队 | 中（每周2-5次） | 低 |
| **新手父母** | 28-40岁，带0-6岁小孩出行 | 需要母婴台、干净、安全，普通厕所无法满足 | 高（每次出门都需要） | 中高 |
| **孕期女性** | 25-38岁 | 频繁如厕需求，需要坐便、干净、不排队 | 高（孕中晚期每天多次） | 中 |
| **老年群体** | 60岁以上，有出行需求 | 需要无障碍设施、坐便，怕楼梯和蹲坑 | 中 | 低 |
| **外卖/快递骑手** | 20-40岁，全天在外 | 熟知城市厕所分布，需要快速可用，是天然的数据贡献者 | 极高（每天多次） | 低 |
| **旅游人群** | 全年龄段，在陌生城市 | 完全不知道哪里有厕所，语言不通时更困难 | 低（出行期间高频） | 低 |
| **肠胃敏感人群** | 全年龄段，IBS/乳糖不耐等 | 突发腹泻，需要最快最近的可用厕所+确保有纸 | 不定（突发高频） | 中 |

## 三、需求价值评估

### 3.1 需求价值维度

| 维度 | 评估 | 说明 |
|------|------|------|
| **用户价值** | ⭐⭐⭐⭐⭐ | 刚需高频，解决真实痛点。几乎人人都有过"急得找不到厕所"的经历 |
| **商业价值** | ⭐⭐⭐ | 短期变现路径有限，但用户基数大。中长期可探索：本地生活广告（商场/餐厅付费推荐厕所）、数据服务（向地图公司/政府出售厕所状态数据）、会员增值（更多场景模式） |
| **竞争壁垒** | ⭐⭐⭐⭐⭐ | 社区众包数据 + 空间关系知识图谱 + 场景推荐算法，三重壁垒。地图公司无法在厕所这个细分品类建立社区 |
| **可实现性** | ⭐⭐⭐⭐ | 微信小程序形态，技术复杂度可控。核心难点在冷启动数据，不在技术 |
| **传播性** | ⭐⭐⭐⭐⭐ | "急厕"这个名字本身就具有传播力。微信分享"这个厕所有纸"是天然社交传播场景 |

### 3.2 影响范围

| 影响维度 | 范围 |
|----------|------|
| 用户端 | 微信小程序，无需下载，即用即走 |
| 数据端 | 需要种子数据录入 + 社区反馈机制 + 开放数据导入 |
| 技术端 | 小程序前端 + 后端API + AI推荐引擎 + 数据管理后台 |
| 运营端 | 种子用户招募 + 数据质量审核 + 社区治理 |
| 合规端 | 地图数据使用授权（腾讯/高德）、用户隐私、内容审核 |

## 四、需求池与优先级建议

### 4.1 完整需求池

按「P0 = MVP必须 / P1 = V1.1快速迭代 / P2 = V2规划 / P3 = 远期展望」分级：

#### P0 — MVP核心（首版必须上线）

| 编号 | 需求 | 说明 | 复杂度 | 依赖 |
|------|------|------|--------|------|
| P0-01 | 一键AI推荐 | 点击"急"按钮，自动推荐1个最佳厕所 | 中 | 推荐算法、厕所数据 |
| P0-02 | 厕所基础数据录入 | 覆盖试点区域200-400个核心厕所的空间参照物、准入条件、设施信息 | 高（人工） | 种子用户 |
| P0-03 | 三色状态灯 | 绿/黄/红实时可用状态 + 时效衰减 + 可信度展示 | 中 | 反馈数据 |
| P0-04 | 轻量选择题反馈 | 3秒完成：能用/不能用 + 干净程度 + 排队 + 有纸 | 低 | 无 |
| P0-05 | 三种场景模式 | 小解急（默认）/ 腹泻急 / 带娃急 | 中 | 推荐算法适配 |
| P0-06 | 质量画像展示 | 清洁度/排队/设施/纸巾 四个维度的选择题分布 | 低 | 反馈数据 |
| P0-07 | 非公厕准入标注 | 麦当劳/星巴克/商场/酒店等商业厕所的准入类型和用户实际体验 | 中 | 数据录入 |
| P0-08 | 导航终点=入口 | 导航到厕所入口而非建筑中心，含"海底捞旁"等文字指引 | 中 | 空间参照物数据、地图API |
| P0-09 | "帮我确认"分享 | 生成厕所链接分享给朋友，接收者可一键确认状态 | 低 | 无 |
| P0-10 | 微信小程序框架 | 首页、推荐结果页、反馈页、分享页 | 中 | 无 |

## 五、MVP 最小可行产品定义

### 5.1 MVP 产品边界

**做什么：**
- 微信小程序，单一城市试点
- 覆盖200-400个核心厕所（地铁站+商圈+公园+商业厕所）
- 一键推荐最优厕所
- 导航到入口（含文字指引）
- 3秒选择题反馈
- 三种场景模式（小解/腹泻/带娃）

**不做什么（V1不做）：**
- 用户注册/登录系统（先用微信OpenID匿名）
- 用户积分/等级
- 历史记录/收藏
- 照片/文字评价
- 离线数据包
- 多城市
- 商业合作

### 5.2 MVP 核心用户流程

```
用户打开小程序
    │
    ├── 首页："急！找厕所"大按钮
    │       │
    │       ├── 默认 = 小解急模式
    │       ├── 长按 = 展开场景选择
    │       └── 语音 = "我拉肚子" → 腹泻模式
    │
    ▼
AI推荐结果页（约2-3秒加载）
    │
    ├── 🥇 最佳推荐（大字卡片）
    │   ├── 名称 + 入口描述
    │   ├── 步行时间 + 可用状态 + 准入条件
    │   └── 质量标签（干净👍 排队少👍 有纸👍）
    │
    ├── 备选列表（下滑查看）
    │
    └── [ 开始导航 ] → 跳转微信地图导航
                         终点 = 厕所入口坐标
                         附带文字指引
    │
    ▼
用户到达厕所 → 使用完毕
    │
    ▼
反馈弹窗（轻量，3秒完成）
    ├── [ ✅ 能用 ]  [ ❌ 不行 ]
    ├── 干净：愿意多待 / 能用 / 不太行
    ├── 排队：没人 / 1-2人 / 排长队
    └── 有纸：有 / 快没了 / 没有
```

### 5.3 MVP 技术方案建议

| 层面 | 方案 | 理由 |
|------|------|------|
| 前端 | 微信小程序原生 | 零安装、微信生态内分享传播、地图能力内置 |
| 地图 | 腾讯位置服务（微信内置） | 小程序原生支持，无需额外SDK |
| 后端 | Node.js / Python + 云函数 | 轻量、快速迭代 |
| 数据库 | 云数据库（MongoDB/PostgreSQL） | 灵活的数据结构，支持地理位置查询 |
| AI推荐 | 规则引擎（V1）→ ML模型（V2） | MVP用规则引擎可控可解释，V2引入机器学习 |
| 部署 | 微信云开发 / 腾讯云 | 降低运维成本 |

### 5.4 MVP 数据模型概要

```
厕所 (toilet)
├── id
├── name
├── type: 公厕 | 商场厕所 | 地铁厕所 | 餐饮厕所 | ...
├── location: { lat, lng }
├── entry_point: { lat, lng }          ← 导航终点
├── spatial_refs: [                      ← 空间参照物
│     { type: "landmark", value: "海底捞旁", floor: "3F" },
│     { type: "direction", value: "3号电梯上3楼，出来右转" }
│   ]
├── access_type: 自由进入 | 需安检 | 建议消费 | ...
├── facilities: {                        ← 设施清单
│     has_baby_station, has_accessible, has_squat, has_seat,
│     has_paper, has_soap, has_hand_dryer
│   }
├── status: {                            ← 实时状态（众包更新）
│     color: green | yellow | red,
│     last_feedback_time,
│     recent_votes: { available: N, unavailable: N }
│   }
└── quality: {                           ← 质量画像（选择题分布）
│     cleanliness: { good: N, ok: N, bad: N },
│     queue: { empty: N, few: N, long: N },
│     paper: { yes: N, low: N, no: N }
│   }
```

## 六、关键风险与应对策略

| 风险 | 严重程度 | 发生概率 | 应对策略 |
|------|----------|----------|----------|
| **冷启动数据不足** | 🔴 高 | 🔴 高 | 种子用户人工录入200-400个核心厕所；外卖骑手合作；开放数据导入(OSM)；"帮我确认"社交裂变 |
| **反馈数据稀疏** | 🟡 中 | 🟡 中 | 到达检测自动弹窗；反馈极致轻量(3秒)；展示"你帮了下一个人"的正向反馈 |
| **数据质量/恶意反馈** | 🟡 中 | 🟡 中 | 隐名信任评分系统；位置校验；多人交叉验证；异常模式检测 |
| **地图数据版权** | 🟡 中 | 🟢 低 | 使用微信内置腾讯地图（合规）；自建空间参照物数据（不依赖第三方POI） |
| **小程序审核** | 🟡 中 | 🟡 中 | 名称可能敏感（"急厕"）；内容审核机制；提前准备替代名称 |
