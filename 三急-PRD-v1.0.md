## 一、背景

### 1.1 问题定义

用户在外出时突然需要如厕，打开地图App搜"厕所"，面临三个地图无法回答的问题：

| 问题 | 地图App的回答 | 用户真实需要 |
|------|-------------|-------------|
| 厕所在哪？ | 建筑中心坐标 | 厕所入口在哪（"3楼海底捞旁"） |
| 能用吗？ | 无法回答 | 实时可用状态 |
| 哪个最好？ | 按距离排序 | 场景化推荐（腹泻要纸、带娃要母婴台） |

**核心痛点**：地图App解决的是"POI搜索"，用户需要的是"资源调度决策"。

### 1.2 市场背景

- 2024年中国公厕改造市场规模达587亿元，但投入集中在硬件改造，软件层面近乎空白
- 现有竞品（高德/百度地图、便了么、今厕一定得、AI厕急达）均停留在"静态POI搜索"阶段，无实时状态、无场景推荐、无空间参照物
- 成都市常住人口2147.4万，外来人口占30%，川菜辛辣文化使腹泻场景高频，是验证产品的天然试验田

### 1.3 产品定位

**三急不是"更好的厕所搜索"，是"厕所实时资源调度引擎"。**

| 维度 | 地图App | 三急 |
|------|---------|------|
| 解决什么问题 | 厕所在哪 | 我应该去哪个 |
| 数据模型 | POI（静态位置） | Resource（动态资源） |
| 排序逻辑 | 直线距离 | 到达时间 × 可用概率 × 质量适配 × 场景匹配 |
| 用户行为 | 搜索→选一个→导航 | 一键→系统推荐→导航 |
| 数据来源 | 商业采集 | 用户众包 + 种子录入 |
| 护城河 | POI数量 | 实时状态数据 + 空间关系知识图谱 |

## 二、目标

### 2.1 产品目标

| 目标 | 衡量指标 | 目标值 |
|------|----------|--------|
| 用户能快速找到合适的厕所 | 首页到推荐结果耗时 | ≤3秒 |
| 推荐结果准确可用 | 用户到达推荐厕所后确认"能用"的比例 | ≥80% |
| 数据飞轮运转 | 每日新增反馈数 | ≥50条/日（上线1个月内） |
| 社交传播 | 分享链接打开率 | ≥30% |

### 2.2 业务目标

| 目标 | 衡量指标 | 目标值 |
|------|----------|--------|
| 验证产品价值 | 成都试点区域覆盖厕所数 | 150-200个 |
| 获取种子用户 | 上线1个月活跃用户 | ≥1000 |
| 验证数据飞轮 | 反馈转化率（到达→反馈） | ≥15% |

### 2.3 非目标（V1明确不做）

- 用户注册/登录系统（用微信OpenID匿名）
- 用户积分/等级系统
- 历史记录/收藏
- 照片/文字评价
- 离线数据包
- 多城市扩展
- 语音输入（移至V1.1）
- 商业化变现

## 三、用户故事

### 3.1 核心用户故事

| 编号 | 用户故事 | 场景 |
|------|----------|------|
| US-01 | 作为一个**逛街的上班族**，我想要**一键找到最近的可用厕所**，这样我不用在街上急着找 | 逛街尿急（最高频） |
| US-02 | 作为一个**吃火锅后腹泻的人**，我想要**快速找到一个确定有纸的厕所**，这样我不会陷入尴尬 | 腹泻急（硬约束：有纸） |
| US-03 | 作为一个**带幼儿出行的父母**，我想要**找到一个有母婴台且干净的厕所**，这样我能给孩子换尿布 | 带娃急（硬约束：母婴台） |
| US-04 | 作为一个**不熟悉成都的游客**，我想要**精确导航到厕所入口而不是建筑大门**，这样我不会在商场里迷路 | 旅游场景 |
| US-05 | 作为一个**使用过厕所的用户**，我想要**用3秒完成反馈**，这样我能帮到下一个急的人 | 数据飞轮 |
| US-06 | 作为一个**收到朋友分享的厕所链接的人**，我想要**一键打开就能导航**，这样我不用下载任何东西 | 社交裂变 |
| US-07 | 作为一个**种子数据贡献者**，我想要**用标准化表单录入厕所信息**，这样我的贡献能被结构化使用 | 冷启动 |

#### F-01 首页极简界面

**描述**：首页展示全屏地图，底部居中放置1个"急"大按钮。遵循Apple HIG的Clarity和Deference原则。

**EARS需求**：

| 类型 | 需求描述 |
|------|----------|
| **Ubiquitous** | 系统应始终在首页展示全屏地图，地图以上方区域为内容主体，用户位置标记居中显示 |
| **Ubiquitous** | 系统应始终在首页底部居中展示1个"急"按钮，按钮直径不小于屏幕宽度的40% |
| **Event-driven** | 当用户点击"急"按钮时，系统应触发智能急推荐流程（F-02） |
| **Event-driven** | 当用户长按"急"按钮（≥500ms）时，系统应展开场景选择面板，展示"腹泻急"和"带娃急"两个选项 |
| **State-driven** | 当用户未授权位置信息时，系统应在地图中央展示"点击定位"引导按钮，"急"按钮置灰不可点击 |
| **Unwanted** | 如果获取用户位置失败，系统应展示"定位失败，请检查GPS设置"提示，并提供"重试"按钮 |

**交互细节**：
- 地图自动缩放至用户当前位置，默认展示半径500米范围
- 地图上展示附近已录入厕所的标记点（自定义图标）
- "急"按钮文案为"急"，下方副文案"长按选场景"
- 长按展开的场景面板从底部弹出（Sheet式），包含2个选项卡片

---

#### F-02 智能急推荐（默认模式）

**描述**：用户点击"急"按钮后，系统根据当前位置、时间、历史偏好，综合计算推荐1个最优厕所。

**EARS需求**：

| 类型 | 需求描述 |
|------|----------|
| **Event-driven** | 当用户点击"急"按钮且位置已授权时，系统应在3秒内完成推荐计算并展示结果 |
| **State-driven** | 系统计算推荐时，应基于以下权重矩阵综合评分：推荐分数 = (1/到达时间) × 可用概率 × 质量适配度 × 场景匹配度 |
| **State-driven** | 智能急模式下，系统应根据上下文信号自动调整权重：时间=夜间(22:00-6:00)时安全区权重↑；时间=饭点时排队权重↑；位置=地铁站时步行时间额外+2分钟(安检) |
| **Ubiquitous** | 系统应仅展示1个最佳推荐结果在首屏，备选结果通过上滑查看 |
| **Unwanted** | 如果搜索半径500米内无任何已录入厕所，系统应自动扩大搜索半径至1000米，并提示"附近厕所较少，已扩大搜索范围" |
| **Unwanted** | 如果扩大至1000米仍无结果，系统应展示"当前区域暂无数据，欢迎贡献"并提供"贡献厕所"入口 |

**推荐算法权重矩阵（智能急模式）**：

| 因子 | 权重 | 计算方式 |
|------|------|----------|
| 到达时间 | 40% | 步行距离 + 过马路(+30s) + 室内额外时间(+2min) + 安检时间(+2min) |
| 可用概率 | 30% | 基于反馈时效衰减计算（见F-09） |
| 质量适配度 | 20% | 清洁度分布 + 排队情况 + 设施匹配 |
| 场景匹配度 | 10% | 智能急模式下根据时间/位置动态调整 |

---

#### F-03 腹泻急模式

**描述**：用户长按"急"按钮选择"腹泻急"后，系统以"有纸"为硬筛选条件，优先推荐有纸+隐私性高+快速的厕所。

**EARS需求**：

| 类型 | 需求描述 |
|------|----------|
| **Event-driven** | 当用户选择"腹泻急"模式时，系统应将"有纸"设为硬筛选条件，排除所有无纸或纸巾状态未知的厕所 |
| **State-driven** | 腹泻急模式下，推荐权重应调整为：到达时间50% + 纸巾保证40% + 隐私性10% |
| **State-driven** | 腹泻急模式下，系统应优先推荐隔间多、人流少的厕所，隐私性权重提升 |
| **Ubiquitous** | 推荐结果卡片应标注"腹泻优选"标签，让用户知道推荐逻辑 |

**腹泻急权重矩阵**：

| 因子 | 权重 | 说明 |
|------|------|------|
| 到达时间 | 50% | 速度仍最重要 |
| 纸巾保证 | 40% | 硬筛选：有纸才能入选；优选"有纸"票数高的 |
| 隐私性 | 10% | 优先隔间多、人流少的 |

---

#### F-04 带娃急模式

**描述**：用户长按"急"按钮选择"带娃急"后，系统以"有母婴台"为硬筛选条件，优先推荐干净+有母婴台的厕所。

**EARS需求**：

| 类型 | 需求描述 |
|------|----------|
| **Event-driven** | 当用户选择"带娃急"模式时，系统应将"有母婴台"设为硬筛选条件，排除所有无母婴台的厕所 |
| **State-driven** | 带娃急模式下，推荐权重应调整为：清洁度60% + 有母婴台(硬筛选) + 到达时间20% + 安全区20% |
| **State-driven** | 带娃急模式下，系统应排除偏僻/停车场/夜间人少的位置 |
| **Ubiquitous** | 推荐结果卡片应标注"带娃优选"标签 |

**带娃急权重矩阵**：

| 因子 | 权重 | 说明 |
|------|------|------|
| 清洁度 | 60% | 干净最重要 |
| 有母婴台 | 硬筛选 | 没有直接排除 |
| 到达时间 | 20% | 距离可容忍远一点 |
| 安全区 | 20% | 排除偏僻位置 |

#### F-05 推荐结果卡片

**描述**：推荐结果以底部卡片形式展示（参照Apple Maps），包含厕所名称、入口描述、步行时间、可用状态、准入条件、质量标签。

**EARS需求**：

| 类型 | 需求描述 |
|------|----------|
| **Ubiquitous** | 推荐结果卡片应始终展示以下信息：厕所名称、入口描述（如"IFS 3F 海底捞旁"）、步行时间、三色状态灯、准入类型、质量标签 |
| **Ubiquitous** | 卡片底部应始终展示"开始导航"主按钮，按钮颜色为橙红色(#FF6B35) |
| **Event-driven** | 当用户点击卡片非按钮区域时，系统应展开卡片展示详细信息（完整空间指引、设施清单、质量画像分布） |
| **State-driven** | 当推荐结果为最佳推荐时，卡片左上角应展示"🥇 最佳推荐"标识 |

**卡片信息层级**：

```
┌─────────────────────────────────┐
│ 🥇 最佳推荐                      │
│ IFS 3F 海底捞旁                  │  ← 主标题
│ 步行3分钟 · 🟢 5分钟前可用        │  ← 核心信息
│ 🆓 免费进入                      │  ← 准入条件
│ 干净👍 排队少👍 有纸👍            │  ← 质量标签
│ [        开始导航 →        ]    │  ← 主按钮
└─────────────────────────────────┘
```

**EARS需求**：

| 类型 | 需求描述 |
|------|----------|

| **Event-driven** | 当用户点击某个备选条目时，系统应将其切换为当前推荐结果，卡片展开为完整信息 |

#### F-07 精准导航到入口

**描述**：用户点击"开始导航"后，系统跳转至微信内置地图展示终点（厕所入口坐标），用户可选择外部地图App导航。

**EARS需求**：

| 类型 | 需求描述 |
|------|----------|
| **Event-driven** | 当用户点击"开始导航"时，系统应调用`wx.openLocation`，传入厕所入口坐标(entry_point)而非建筑中心坐标(location) |
| **Ubiquitous** | `wx.openLocation`的name参数应传入厕所名称，address参数应传入入口描述（如"3号电梯上3楼，出来右转"） |
| **State-driven** | 当厕所为室内厕所（有楼层信息）时，系统应在导航启动前展示一个覆盖层提示"导航至建筑入口，到达后请查看楼层指引" |
| **Unwanted** | 如果`wx.openLocation`调用失败，系统应展示"导航启动失败，请重试"并提供重试按钮 |

#### F-08 到达后文字指引

**描述**：用户从外部地图App返回小程序时（或导航完成后），展示厕所的详细路径指引。

**EARS需求**：

| 类型 | 需求描述 |
|------|----------|
| **Event-driven** | 当用户从导航返回小程序时，系统应展示该厕所的完整路径指引覆盖层 |
| **Ubiquitous** | 路径指引应包含：锚点品牌（如"海底捞"）、楼层（如"3F"）、方向指引（如"3号电梯上3楼，出来右转，海底捞门口旁通道进入"） |

│                                 │

#### F-09 三色状态灯

**描述**：每个厕所展示实时可用状态，以三色信号灯+形状+文字呈现，基于社区反馈时效衰减计算。

**EARS需求**：

| 类型 | 需求描述 |
|------|----------|
| **Ubiquitous** | 系统应始终为每个厕所展示三色状态灯之一：🟢可用（绿色圆形●）、🟡存疑（黄色三角形▲）、🔴不可用（红色方形■） |
| **State-driven** | 系统应根据以下规则计算状态：<br>• 最近5分钟内有"可用"反馈 → 🟢 可用，可信度≥90%<br>• 最近1小时内有"可用"反馈但超过5分钟 → 🟢 可用，可信度60-90%<br>• 最近反馈超过1小时但不超过3小时 → 🟡 存疑，可信度40-60%<br>• 最近反馈超过3小时，或最近反馈为"不可用" → 🟡 或 🔴 |
| **State-driven** | 当有用户报告"维修中/不可用"且1小时内有≥3人确认不可用时，系统应展示🔴不可用 |
| **Event-driven** | 当用户点击状态灯时，系统应展示状态详情：最近反馈时间、今日确认人数、可信度百分比 |
| **State-driven** | 系统应根据厕所类型应用不同的时效衰减规则：<br>• 商场厕所：营业时间外所有反馈降为🟡<br>• 地铁厕所：末班车后反馈无效<br>• 24小时公厕：反馈24小时后开始衰减 |

**状态计算逻辑**：

```
状态 = f(最近反馈时间, 反馈方向, 反馈数量, 厕所类型, 运营时间)

🟢 可用（高可信）：
  - 最近5分钟内有"可用"反馈
  - 可信度 = 95%

🟢 可用（中可信）：
  - 最近1小时内有"可用"反馈，但超过5分钟
  - 可信度 = 60-90%，随时间衰减

🟡 存疑：
  - 最近反馈超过1小时但不超过3小时
  - 或：最近反馈超过3小时，但无"不可用"报告
  - 可信度 = 40-60%

🔴 不可用：
  - 最近1小时内有≥3人报告"不可用"
  - 或：最近反馈为"不可用"且可信度≥85%

特殊规则：
  - 商场厕所闭店后 → 自动降为🟡
  - 地铁末班车后 → 自动降为🟡并标注"运营结束"
  - 超过6个月无反馈 → 标注"数据可能过期"
```

#### F-10 3秒选择题反馈

**描述**：用户使用厕所后，通过4道选择题完成反馈，驱动数据飞轮。

**EARS需求**：

| 类型 | 需求描述 |
|------|----------|
| **Event-driven** | 当用户点击"我到了"按钮后，系统应在用户从外部地图返回小程序时，检测用户位置与厕所的距离，若≥100米则弹出反馈Sheet（注：小程序后台无法检测位置变化，改为返回时检测） |
| **Event-driven** | 当用户通过分享链接进入并使用后，系统同样应在用户返回小程序时弹出反馈 |
| **State-driven** | 当用户连续跳过反馈2次后，系统应将反馈弹窗频率降低为每3次到达弹1次，避免骚扰 |
| **Ubiquitous** | 反馈Sheet应包含4道选择题：<br>1. 这个厕所：[✅能用] [❌不行]<br>2. 干净程度：[愿意多待] [能用] [不太行]<br>3. 排队情况：[没人] [1-2人] [排长队]<br>4. 有纸吗：[有] [快没了] [没有] |
| **Event-driven** | 当用户提交反馈时，系统应展示"你帮了下一个急的人 🤝"正向确认，并关闭Sheet |
| **Event-driven** | 当用户点击"跳过"时，系统应直接关闭Sheet，不强制反馈 |
| **Unwanted** | 如果用户位置与厕所距离≥500米时尝试反馈，系统应将反馈权重降低（位置不匹配，可能虚假） |

#### F-15 数据审核后台

**描述**：管理员通过Web后台审核用户提交的厕所数据，查看反馈数据看板。

**EARS需求**：

| 类型 | 需求描述 |
|------|----------|

## 七、数据指标

### 7.1 核心指标

| 指标 | 定义 | 目标值 | 埋点事件 |
|------|------|--------|----------|
| 首页到推荐耗时 | 从打开小程序到推荐结果展示的时间 | ≤3秒 | `app_launch` → `recommend_result_show` |
| 推荐采纳率 | 用户点击"开始导航"的次数 / 推荐结果展示次数 | ≥60% | `recommend_result_show` / `nav_start` |

| 分享率 | 分享次数 / 推荐结果展示次数 | ≥5% | `recommend_result_show` / `share_click` |
| 分享打开率 | 分享链接被打开次数 / 分享次数 | ≥30% | `share_click` / `share_open` |

| 指标 | 定义 | 目标值 |

| 日均反馈数 | 每日新增反馈条数 | ≥50（上线1个月内） |
| 数据新鲜度 | 厕所最近一次反馈距今时间的中位数 | ≤24小时 |

## 九、附录：边界场景与异常处理

### 9.1 边界场景

| 场景 | 处理方式 |
|------|----------|
| 用户在隧道/地下停车场GPS弱 | 提示"定位信号弱，请走到开阔处"；允许手动选择附近区域 |
| 厕所搜索范围内无结果 | 自动扩大至1000米；仍无结果提示"欢迎贡献" |
| 所有厕所状态均为🔴不可用 | 推荐最近的🟡存疑厕所，标注"附近厕所可能不可用，建议谨慎" |
| 商场厕所闭店时间 | 自动降为🟡；推荐时标注"可能已闭店" |
| 地铁末班车后 | 标注"运营结束"；推荐时排除或降权 |
| 用户多次跳过反馈 | 不强制；降低反馈弹窗频率（每3次到达弹1次） |
| 反馈与实际严重不符（可信度低的用户） | 系统降低该用户信任分，其反馈权重降低 |
| 室内厕所楼层指引不完整 | 展示已有指引 + 标注"指引不完整，欢迎补充" |

### 9.2 异常状态处理

| 异常 | 处理方式 |
|------|----------|
| 定位失败 | 展示"定位失败"提示 + 重试按钮；允许手动输入位置 |
| 网络断开 | 展示"网络异常"提示；缓存最近一次推荐结果（可查看但不可刷新） |
| 服务器错误 | 展示"服务繁忙"提示 + 重试按钮 |
| wx.openLocation调用失败 | 展示"导航启动失败" + 重试；备选方案：展示厕所坐标和入口文字描述，引导用户手动打开地图App |
| 审核后台无法访问 | 提示"无权限"或"服务异常"；不影响小程序端正常使用 |

### 9.3 权限规则

| 角色 | 可看 | 可操作 |
|------|------|--------|
| 未授权位置的游客 | 首页地图（无定位） | 授权定位后才可点击"急"按钮 |
| 普通用户（L1） | 所有已上线厕所信息 | 推荐导航、反馈状态、提交厕所、提交纠错、分享 |
| 城市审核员（L2） | 审核后台本市数据 | 审核新提交、查看反馈看板 |
| 大区管理员（L3） | 审核后台全量数据 | 锁定厕所、审核争议、编辑热门厕所 |
| 平台运营（L4） | 全部数据 | 处理申诉、封禁恶意账号、数据策略配置 |

### 9.4 数据口径

| 数据 | 口径定义 |
|------|----------|
| "可用"反馈 | 用户在厕所范围(≤500m)内提交的"✅能用"反馈 |
| "不可用"反馈 | 用户提交的"❌不行"反馈，需≥3人/小时内交叉验证才降为🔴 |
| 可信度 | 基于反馈者信任分 × 时间衰减系数 × 位置匹配度计算 |
| 信任分 | 新用户初始60分；反馈被其他用户交叉验证一致 → +5；不一致 → -5；范围0-100 |
| 质量画像时间窗 | 最近10条反馈，按时间加权（越近权重越高） |
| 覆盖厕所数 | 已审核上线状态的厕所数量（不含待审核和已下线） |

## 十一、评审决策记录（v1.1更新）

| # | 问题 | 决策 | 决策依据 |
|---|------|------|----------|
| 1 | 后端数据库选型 | **PostgreSQL + PostGIS** | 地理查询能力强，支持ST_DWithin精确查询 |
| 2 | 推荐算法缓存 | **引入Redis** | 热门厕所状态缓存5分钟，减少数据库压力 |
| 3 | 反馈弹窗触发机制 | **返回小程序时检测距离** | 小程序后台无法检测位置变化，改为返回时检测≥100m弹出 |
| 4 | 地图个性化样式 | **MVP用默认样式** | 个性化样式需付费，V1.1再购入 |
| 5 | 审核后台形态 | **独立Web应用** | 功能完整可扩展，不受小程序限制 |
| 6 | 状态计算策略 | **热门缓存+冷门实时** | 热门厕所缓存5分钟，冷门实时计算 |
| 7 | 多次跳过反馈 | **每3次到达弹1次** | 连续跳过2次后降频，避免骚扰 |
| 8 | 地图标记点击 | **先简卡片再进详情** | 减少跳转层级，底部简卡片→点击进详情 |
| 9 | 分享卡片文案 | **质量标签驱动** | "三急·IFS 3F·🟢·有纸"，比步行时间更有吸引力 |
| 10 | 数据过期状态 | **复用🟡+文字标注** | 不增加独立状态，控制设计复杂度 |
