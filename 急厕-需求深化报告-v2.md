## 一、产品名称候选分析

### 1.1 命名设计原则

社交裂变场景下，名称需同时满足：

| 原则 | 说明 | 验证方法 |
|------|------|----------|
| **不直白** | 不含"厕/厕所/卫生间"等词，避免分享尴尬 | 微信聊天发出去不脸红 |
| **可联想** | 用户看到名字能大致猜到功能 | 不需要过多解释 |
| **有记忆点** | 有梗、有典故或有反差，便于口口相传 | "你知道吗XX其实是这个意思" |
| **好传播** | 2-3字，好念好打，无生僻字 | 口语说出来顺嘴 |
| **可注册** | 小程序名称、商标未被占用 | 微信小程序搜索验证 |

### 1.2 社交裂变模拟测试

以微信聊天场景验证名称的社交友好度：

```
场景A — 微信私聊分享
  "我用【XXX】找到的，IFS 3楼海底捞旁边，有纸"
  → XXX 是否让发送者尴尬？接收者是否想点开？

场景B — 朋友圈
  "【XXX】这个小程序救了我一命 lol"
  → XXX 是否适合公开展示？

场景C — 小程序分享卡片
  卡片标题：「XXX — IFS 3楼厕所 · 步行3分钟 · 有纸」
  → XXX 是否引发好奇？
```

### 1.3 候选方案

#### 方案A：三急（sān jí）

| 维度 | 分析 |
|------|------|
| **典故** | "人有三急"——尿急、便急、屁急，出自《论语》典故延伸，人人皆知 |
| **联想** | 不直说厕所，但人人都懂。文化典故自带认知基础 |
| **社交测试** | "我用三急找到的" → 自然、不尴尬、略带幽默 |
| **传播话题** | "人有三急是哪三急你知道吗？" → 天然的社交话题入口 |
| **优点** | 两字简洁；典故认知度高；不直白但指向明确；有幽默感 |
| **风险** | "三急"可能已被注册为小程序名；含义稍泛 |
| **推荐度** | ⭐⭐⭐⭐⭐ |

#### 方案B：急所（jí suǒ）

| 维度 | 分析 |
|------|------|
| **典故** | 围棋术语，指棋局中急需处理的要害位置。双关：急+地方 |
| **联想** | 需要一定门槛理解。但"急"+"所"字面也可解读为"着急的地方" |
| **社交测试** | "用急所找到的" → 略显生硬，需要解释才知道是什么 |
| **传播话题** | "围棋里有个术语叫急所，现在有个找厕所App叫这个" → 高话题性 |
| **优点** | 文雅、抽象、有文化深度；双关精妙；高话题性适合内容营销 |
| **风险** | 认知门槛高；非围棋爱好者可能完全get不到；可能被认为故弄玄虚 |
| **推荐度** | ⭐⭐⭐⭐ |

#### 方案C：解手（jiě shǒu）

| 维度 | 分析 |
|------|------|
| **典故** | 四川方言"解手"=上厕所。源自"湖广填四川"历史典故——移民被绑押送，如厕需"解手"，遂成方言 |
| **联想** | 四川/西南地区人人皆知；全国范围也有一定认知度（文学作品常见） |
| **社交测试** | "解手帮我找到了" → 成都极其自然；有地域亲切感 |
| **传播话题** | "你知道'解手'这个词怎么来的吗？" → 历史典故自带传播力 |
| **优点** | **与成都试点完美契合**；有历史典故深度；文雅不直白；地域文化感强 |
| **风险** | 非西南地区用户可能不懂；全国扩展时认知门槛上升 |
| **推荐度** | ⭐⭐⭐⭐ |

#### 方案D：方便了（fāng biàn le）

| 维度 | 分析 |
|------|------|
| **典故** | "方便"=如厕的委婉说法；"了"=完成/找到。双关："找到方便之所了" |
| **联想** | 口语化，指向明确但不直白 |
| **社交测试** | "方便了帮我找到了" → 最轻松、零社交负担 |
| **传播话题** | 弱，过于日常，缺乏记忆点 |
| **优点** | 零社交负担；双关精妙；最口语化 |
| **风险** | 品牌感弱；"方便了"可能已被注册；含义太泛 |
| **推荐度** | ⭐⭐⭐ |

### 1.4 综合推荐

| 排名 | 名称 | 推荐理由 |
|------|------|----------|
| **首选** | **三急** | 典故认知度最高、不直白但人人都懂、有幽默感、传播最自然 |
| **备选** | **急所** | 最文雅抽象、话题性最强、适合内容营销驱动的传播 |
| **地域优选** | **解手** | 与成都试点完美契合、有历史典故、地域文化感强 |

> 💡 **建议**：首选「三急」。如果小程序名已被注册，退而求其次选「急所」。如果希望强化成都地域特色，选「解手」。

## 二、成都试点城市策略

### 2.1 成都为什么是最佳试点城市

| 维度 | 数据 | 与急厕的关联 |
|------|------|-------------|
| **人口规模** | 常住人口2147.4万（2024年），全国第四 | 用户基数大，样本充足 |
| **人口净流入** | 2024年净增7.1万，川渝年净流入近30万，年轻人口流入率高 | 外来人口不熟悉本地厕所分布→需求强 |
| **外来人口占比** | 外来人口639万，占常住人口30% | 大量人需要"找厕所"而非"知道厕所" |
| **美食文化** | 川菜辛辣、火锅串串文化 | 肠胃挑战多→腹泻场景需求真实存在（这是天然的产品验证场景） |
| **商圈密集** | 春熙路、太古里、IFS、宽窄巷子、锦里等 | 室内厕所场景丰富，验证"空间参照物"逻辑 |
| **地铁网络** | 13条线路运营，日均客流400万+ | 地铁厕所是核心场景之一 |
| **公厕建设** | 2017-2020年新建800座+改造1500座；出台《成都市公共厕所管理条例》 | 政府重视，公厕数据基础好 |
| **城市包容性** | "来了就是成都人"；青年人才占比高 | 接受新事物、愿意尝鲜、愿意反馈 |
| **厕所痛点明确** | 2024年媒体报道：金牛区城隍庙社区"公厕数量少、位置偏、如厕难" | 痛点被官方媒体确认，需求真实 |

### 2.2 成都试点区域规划

分三圈层逐步推进：

## 三、基于HIG的设计哲学与场景方案

### 3.1 Apple HIG核心原则在急厕中的应用

| HIG原则 | 含义 | 在急厕中的应用 |
|---------|------|---------------|
| **Clarity（清晰）** | 文字易读、图标精确、装饰适度 | 首页只有1个核心操作；推荐结果用大字+图标，信息一目了然 |
| **Deference（顺从）** | UI服务于内容，不喧宾夺主 | 地图为底，操作浮层在上；不使用花哨动画干扰核心任务 |
| **Depth（层次）** | 合理的视觉层级 | 首页→推荐结果→导航，层级清晰；场景选择通过长按展开，不占首屏 |
| **Progressive Disclosure（渐进式展示）** | 复杂选项按需展开 | 首页只给1个按钮；场景模式通过长按/语音触发，不在首页铺开 |
| **Minimal Input（最少输入）** | 减少用户输入操作 | 默认智能推荐，无需选择场景；反馈用选择题而非打字 |
| **Direct Manipulation（直接操作）** | 手势直接操控内容 | 地图可拖拽缩放；卡片可滑动查看备选 |
| **Feedback（即时反馈）** | 每个操作都有响应 | 点击"急"立即显示加载状态；反馈提交后即时确认 |

### 3.2 场景数量设计

**核心矛盾**：场景越多越精准，但越多越违反HIG的渐进式展示原则。急用场景下用户没有耐心做多选。

**设计决策**：遵循HIG，首页只暴露1个操作，场景通过交互手势按需触发。

| 层级 | 场景 | 触发方式 | HIG原则 |
|------|------|----------|---------|
| **默认层** | 智能急（通用模式） | 点击"急"按钮 | Clarity — 首屏只有1个操作 |
| **扩展层** | 腹泻急 | 长按"急"按钮 → 展开场景选择 | Progressive Disclosure — 按需展开 |
| **扩展层** | 带娃急 | 长按"急"按钮 → 展开场景选择 | Progressive Disclosure — 按需展开 |
| **语音层** | 任意场景 | 语音输入"我拉肚子"/"带小孩" | Minimal Input — 零点击 |

**V1共3个场景**（智能急+腹泻急+带娃急），但首页只展示1个按钮，完全符合HIG。

**为什么不做更多场景？**

| 场景 | V1是否做 | 理由 |
|------|----------|------|
| 女性夜间安全 | ❌ V1.1 | 匿名状态下无法自动判断性别，需用户主动选——增加操作步骤，违反Minimal Input。V1.1可通过"智能急"模式在夜间自动提升安全区权重来隐式覆盖 |
| 老人无障碍 | ❌ V1.1 | 同理，需用户主动选。V1.1加入 |
| 孕妇 | ❌ V1.1 | 可复用带娃模式的逻辑（母婴台+干净+坐便），V1.1单独优化 |

### 3.3 智能急模式（默认）的设计逻辑

智能急不是"不区分场景"，而是**系统根据上下文自动调整权重**：

| 信号 | 系统推断 | 权重调整 |
|------|----------|----------|
| 时间=夜间（22:00-6:00） | 安全需求↑ | 安全区权重提升，排除偏僻位置 |
| 时间=饭点（11:30-13:30 / 18:00-20:00） | 排队概率↑ | 排队权重提升，偏好商场多厕位 |
| 位置=旅游区 | 不熟悉环境 | 质量权重↑，偏好有文字指引的 |
| 位置=地铁站 | 安检成本 | 步行时间额外+2分钟（安检时间） |
| 用户历史=偏好干净 | 质量敏感 | 质量权重↑ |

> 智能急模式让90%的用户不需要选择场景，直接点击就得到合适推荐。只有腹泻和带娃这种"硬约束不同"的场景才需要用户主动切换。

## 四、种子数据录入功能设计

### 4.1 设计目标

种子数据是整个产品的地基。录入功能需要满足：
- **标准化**：不同人录入的数据结构一致
- **低门槛**：非技术人员也能操作
- **可校验**：录入后可审核、纠错
- **可扩展**：后续用户也可贡献数据

### 4.2 录入工具方案

提供两种录入方式，满足不同场景：

| 方式 | 形态 | 适用场景 | 优势 |
|------|------|----------|------|
| **在线表单** | 小程序内"贡献厕所"入口 | 用户随手发现新厕所 | 零安装、可拍照定位 |
| **批量录入模板** | Excel/CSV模板下载+导入 | 种子用户集中录入大量厕所 | 效率高、可离线编辑 |

### 4.3 录入数据字段设计

每个厕所的完整数据结构：

```yaml
# 基本信息
toilet_id: 自动生成
name: "IFS 3F 海底捞旁厕所"          # 命名规则：[建筑名] [楼层] [锚点] 厕所
type: "商场厕所"                      # 公厕|商场厕所|地铁厕所|餐饮厕所|公园厕所|加油站厕所|酒店厕所|医院厕所
location:                             # 建筑坐标（用于地图展示）
  lat: 30.6586
  lng: 104.0817
entry_point:                          # 入口坐标（用于导航终点）
  lat: 30.6588
  lng: 104.0820

# 空间参照物（核心差异化数据）
spatial_refs:
  - type: "landmark"                  # 锚点类型
    value: "海底捞"                    # 锚点名称（选择导航设施完善的品牌）
    floor: "3F"                       # 楼层
  - type: "direction"                 # 路径指引
    value: "3号电梯上3楼，出来右转，海底捞门口旁通道进入"
  - type: "entry_note"                # 入口备注

### 4.4 录入流程设计

```

  ├── 系统校验必填项

### 4.6 数据质量保障

| 机制 | 说明 |
|------|------|
| **双人校验** | 每个厕所由2人独立录入，系统比对差异，差异项标记需人工确认 |
| **审核队列** | 所有录入数据先进入审核队列，管理员审核后上线 |
| **用户纠错** | 任何用户可对已上线厕所提交纠错（如"锚点已搬迁"），管理员审核 |
| **自动校验** | 坐标合理性校验（厕所坐标不应在道路中间）；命名规范校验；必填项校验 |
| **过期标记** | 超过6个月无反馈的厕所自动标记为"数据可能过期，欢迎确认" |

### 5.1 参考产品分析

| 参考产品 | 借鉴点 | 不借鉴 |
|----------|--------|--------|
| **Waze** | 首页1个大按钮的极简设计；社区元素（其他用户反馈）；明快的色彩语言 | 过多装饰元素；游戏化倾向 |
| **Apple Maps** | 底部卡片式信息展示；卡片可滑动展开；优雅的视觉留白；深色/浅色模式 | 复杂的Tab导航；过多功能入口 |
| **Google Maps** | 地图与列表协同；清晰的POI图标体系；路线绘制的视觉层级 | 信息密度过高；功能入口过多 |

### 5.2 设计语言定义

| 维度 | 定义 | 理由 |
|------|------|------|
| **主色调** | 紧急感橙红色 #FF6B35（按钮）+ 信任感深蓝 #1A73E8（地图/链接） | 橙红传达"急"的情绪，蓝色传达可靠感 |
| **状态色** | 绿 #34A853 / 黄 #FBBC04 / 红 #EA4335 | 三色信号灯，同时考虑色盲友好（辅以图标形状区分） |
| **字体** | 系统默认（SF Pro / 苹方） | 遵循HIG，不引入自定义字体 |
| **圆角** | 卡片12pt圆角，按钮8pt圆角 | 参照Apple Maps的卡片风格 |
| **地图样式** | 浅色简约底图，厕所点位用自定义图标 | 避免地图色彩干扰厕所点位识别 |
| **动效** | 仅用于状态转换（加载→结果），不使用装饰性动画 | HIG Deference原则 |

### 调研结论：Waze模式数据审核机制

基于Waze社区治理机制的调研，建议三急采用 **4级角色 + 3层审核** 体系：

**角色层级：**

| 角色 | 获取条件 | 权限 |
|------|----------|------|
| L1 普通用户 | 默认 | 提交厕所信息、反馈状态、提交纠错建议 |
| L2 城市审核员 | ≥30条提交被采纳 | 审核本市新提交的厕所数据 |
| L3 大区管理员 | ≥500条贡献 + 准确率≥90% | 可锁定/编辑热门厕所，审核争议 |
| L4 平台运营 | 内部 | 处理申诉、恶意账号、数据策略 |

**审核流程：**

| 数据类型 | 审核方式 | 参照Waze机制 |
|----------|----------|-------------|
| 新厕所提交 | L1提交 → L2+人工审核 → 上线 | Waze的PUR机制（前30条强制审核） |
| 已上线厕所修改 | 众包确认（多用户标记"还在/已拆"自动更新） | Waze的"There/Not there"众包验证 |
| 热门/争议厕所 | 锁定，仅L3+可改 | Waze的锁级别机制 |
| 实时状态反馈 | 无需审核，按信任评分加权聚合 | Waze的实时路况众包验证 |

**信任机制：**
- 新用户前N条提交强制审核
- 信誉分：基于采纳率、位置匹配度动态调整
- 每日提交配额（防刷，参照Waze L1每天10条限制）
- 恶意提交累计达阈值自动降级或封禁

> 💡 **回答你的问题**：参考Waze模式，V1阶段建议**由产品负责人（你）担任L4平台运营+L3大区管理员**，同时招募2-3名种子用户作为L2城市审核员。随着数据量增长，逐步开放L2/L3权限给高贡献社区用户。

### 调研结论：微信小程序地图技术可行性

**结论：技术方案完全可行，无阻断风险。**

| 功能需求 | 可行性 | 实现方案 | 注意事项 |
|----------|--------|----------|----------|
| 自定义终点坐标导航 | ✅ | `wx.openLocation({latitude, longitude, name, address})` | 微信无内置导航，通过此API拉起外部地图App导航 |
| 自定义标记点(Marker) | ✅ | map组件 `markers` 属性，支持自定义`iconPath` | 每个marker可设不同图标 |
| 文字覆盖层（到达指引） | ✅ | marker的 `callout` 属性（如"上3楼右转海底捞旁"） | 或用`cover-view`页面级覆盖层 |
| 步行路线规划绘制 | ✅ | 腾讯位置服务SDK `qqmapsdk.direction({mode:'walking'})` + polyline绘制 | 需引入腾讯位置服务小程序SDK |
| 地图样式自定义 | ⚠️ 付费 | `layer-style` + `subkey` 个性化地图 | 2023年6月起需购买，MVP可先用默认样式 |
| 获取用户位置 | ✅ | `wx.getLocation({type:'gcj02'})` | 需用户授权 |

**关键技术结论：**
1. 微信小程序**没有内置turn-by-turn导航**，导航通过`wx.openLocation`拉起外部地图App完成
2. 步行路线可在小程序内用polyline绘制（展示用），实际导航跳转外部App
3. 个性化地图样式需付费，MVP阶段建议先用默认样式，V1.1再考虑
4. 腾讯位置服务个人开发者每日1万次免费配额，中小应用足够

**技术方案更新：**

| 层面 | 方案 |
|------|------|
| 地图展示 | 微信小程序map组件 + 自定义markers |
| 步行路线 | 腾讯位置服务SDK direction(walking) + polyline绘制 |
| 精准导航 | `wx.openLocation`拉起外部地图App，终点=入口坐标 |
| 到达指引 | map组件callout或cover-view覆盖层 |
| 地图样式 | MVP用默认样式，V1.1购入个性化样式 |
